# 0x0 Problem

`v8asm disasm` used to crash on some bytenode-generated `.jsc` files. The first guess was that object printing needed more SIGSEGV guards, but gdb showed the process crashed earlier, during recursive object discovery:

```text
Thread 1 "v8asm" received signal SIGSEGV, Segmentation fault.
0x0000555555a300b2 in V8ObjectExplorer::Traverse(...)
#0 V8ObjectExplorer::Traverse(...)
#1 V8ObjectExplorer::Traverse(...)
#2 V8ObjectExplorer::Traverse(...)
#6 do_disasm(...)
```

The crashing objects were entries from `TrustedFixedArray` constant pools, usually read-only strings such as `JSON`, `parse`, or `toUpperCase`. Those addresses looked like valid V8 heap tagged pointers, but they were not valid objects for the current isolate.

# 0x2 Actual Cause

The root cause was not simply missing print recovery. The code serializer sanity checks had been disabled, allowing an incompatible cached-data file to deserialize.

For the same test case, the bytenode cache and a cache generated by current `v8asm` had the same version hash, but different magic, flag hash, and read-only snapshot checksum:

```text
bytenode:
  magic: 0xc0de0688
  version_hash: 0x2b2c7714
  flags_hash: 0x1c7c619b
  read_only_snapshot_checksum: 0xd31c4342

vanilla v8asm:
  magic: 0xc0de0689
  version_hash: 0x2b2c7714
  flags_hash: 0xdc93751f
  read_only_snapshot_checksum: 0x436e38a3
```

The version hash alone is therefore insufficient. `13.6.233.10-node.26` and vanilla `13.6.233.10` can share the same `Version::Hash()` while still having incompatible serializer format details, runtime flags, and read-only heap contents.

When this mismatch is forced through deserialization, the resulting object graph can contain references that do not correspond to valid objects in the current isolate. That is why `Traverse()` crashed on `obj->map()` or while reading constant-pool entries.

# 0x3 Current Fix

`v8asm disasm` now checks cached-data compatibility before deserializing:

- `magic`
- `version_hash`
- `flags_hash`
- `read_only_snapshot_checksum`
- `payload_length`

Default behavior is strict:

```bash
./v8asm disasm bytenode.jsc
```

If the file is incompatible, `v8asm` refuses to deserialize and prints a header report.

For research only, there is an explicit recovery mode:

```bash
./v8asm disasm bytenode.jsc --force-incompatible
```

Only this mode enables best-effort SIGSEGV recovery while discovering and printing objects. The output may be partial, but the tool no longer hides the compatibility mismatch.

# 0x4 Decompiler Notes

Best-effort disassembly can include diagnostic lines such as:

```text
!0x...: segmentfault while discovering object, skipped
```

The Python parser now ignores blank lines and `!` diagnostic lines instead of appending them to the previous object block. This matters because a diagnostic line after a `[String]` object made `V8String.parse()` think the string object had multiple lines.

After the parser fix, the decompile round no longer has process-level segfaults, and the bytenode `try/catch` and `all_features` cases produce level-4 output again.

The level-4 postprocessor now also strips two kinds of bytecode recovery noise:

- dead `ACCU = item` moves left inside already recovered `for...of` loops;
- unused boolean iterator-state registers inside those recovered loop bodies.

It also renames nested `for...of` loop variables when the recovered source
expression uses the same name. This avoids invalid or misleading output such as:

```js
for (const item of arg0) {
  for (const item of item) {
```

and renders it as:

```js
for (const item of arg0) {
  for (const item1 of item) {
```

Finally, pure `ACCU` loads that are overwritten before any later `ACCU` read are
removed at level 4. This is deliberately narrower than global dead-store
elimination: effectful expressions such as calls are kept, and `ACCU` reads in
conditions keep their producer.

Current round impact:

```text
03_for_of_sum: v8asm ACCU lines 4 -> 0, bytenode 2 -> 0
04_nested_loop_if: v8asm 4 -> 0, bytenode 2 -> 0
09_all_features: v8asm 18 -> 2, bytenode 16 -> 4
```

The regression corpus was expanded with common bytecode shapes that were not
covered by the first round:

```text
10_array_index.js      keyed property reads, e.g. values[0] and values[i]
11_object_mutation.js  SetNamedProperty, e.g. obj.count = obj.count + 1
12_logical_nullish.js  ??, &&, || control-flow bytecode
```

Those tests exposed missing translator support for `GetKeyedProperty` and
`SetNamedProperty`. After adding those opcodes and a narrow level-4 compaction
pass, the array-index case now recovers:

```js
return (arg0[0] + arg0[arg1])
```

and the object-mutation case recovers:

```js
arg0.count = (arg0.count + 1)
arg0.seen = true
return arg0.count
```

The logical/nullish case also exposed an incorrect simplification: `r0 = ACCU`
was treated as a safe inline alias, which could turn `r0.value` into
`ACCU.value`. `ACCU` is now excluded from single-use register inlining. The
remaining work there is high-level recovery for `??`, `&&`, and `||`; the
current output keeps the lower-level ACCU/control-flow form rather than
inventing a less certain expression.

That follow-up is now implemented as a separate level-4 logical recovery module
instead of adding more rules to the already-large `postprocess.py`. The
postprocess layer was split so the base file remains a small register/ACCU
simplifier, while level-4 rules live in focused modules:

```text
postprocess.py                   base propagation and public entry point
postprocess_level4.py            level-4 pipeline ordering
postprocess_level4_arrays.py     array literal/spread builder recovery
postprocess_level4_common.py     block and register-use helpers
postprocess_level4_forof.py      for...of recovery and cleanup
postprocess_level4_guards.py     iterator/pending-message guard cleanup
postprocess_level4_inline.py     conservative single-use register inlining
postprocess_level4_logical.py    ??, &&, || recovery
postprocess_level4_defaults.py   undefined/default-parameter recovery
postprocess_level4_properties.py keyed/named property compaction
postprocess_level4_binary.py     ACCU binary expression cleanup
postprocess_level4_calls.py      bound .call(...) rewriting
postprocess_level4_strings.py    string concatenation chains
postprocess_level4_switch.py     switch-shaped control flow
```

`12_logical_nullish.js` now recovers the main function body as:

```js
r0 = (isNullish(arg0) ? arg2 : arg0)
if (truthy(r0) && truthy(arg1)) {
  return (r0.value || "missing")
}
return "none"
```

The round summary now shows zero `ACCU` assignment lines for the newly added
`10_array_index`, `11_object_mutation`, and `12_logical_nullish` cases in both
vanilla-v8asm and bytenode modes.

This exposed missing opcode coverage for:

Those opcodes are now translated, and the round analyzer has explicit columns

After the opcode pass, `unknown` is zero across the current corpus. The remaining
non-zero `undefined_fallbacks` are only in bytenode mode, which is expected when
using `--force-incompatible` against Node cached data.

The new cases also found two ACCU propagation bugs:

1. Register propagation must not inline an array/object literal into an
   assignment target. `r8[0] = value` must not become `[][0] = value`.
2. `r0 = ACCU` after a control-flow join cannot be cached as if `ACCU` had a
   single known expression. Otherwise later loads can collapse into
   `ACCU = ACCU`.

Both are fixed in `postprocess.py`, with unit tests. There is one deliberate
trade-off: level 4 now keeps ACCU loads that flow out of an `if/else` block,
because optional chaining carries its selected value through that join. This
raises the ACCU metric on some older cases, but it avoids deleting real values
such as `r2.city` and `r2.length`.

The simple `if/else` case then exposed a smaller expression-cleanup gap. V8 can
load a value into the accumulator and immediately return an accumulator-left
binary expression:

```text
ACCU = arg0
return (ACCU - 10)
```

The binary cleanup already handled accumulator-right shapes such as
`left + ACCU`; it now also handles accumulator-left `+`, `-`, `*`, `/`, and `%`
when the producer is adjacent and does not itself mention `ACCU`:

```js
return (arg0 - 10)
```

This moved `02_if_else` to zero ACCU assignment lines in both vanilla-v8asm and
bytenode modes without changing the recovered control flow.

The destructuring/spread case then exposed an interaction between `for...of`
recovery and array-literal stores. V8 lowers:

```js
[a, b, ...Object.values(rest)]
```

into fixed index stores followed by an iterator loop over `Object.values(rest)`.
The first `for...of` recovery also has to rewrite all uses of
`iteratorResult.value` to the recovered loop item, including direct stores such
as:

```text
r8[r9] = r12.value
```

After that, a focused array-builder pass can collapse:

```text
r8 = []
r8[0] = r0
r8[1] = r1
for (const item of Object.values(r2)) { r8[r9] = item; r9 += 1 }
```

into:

```js
r8 = [r0, r1, ...Object.values(r2)]
```

This moved `13_destructuring_spread` in vanilla-v8asm mode from
`ACCU=4, reg_refs=57` to `ACCU=2, reg_refs=42`. A later default-value cleanup
then handled the `a = 1` destructuring default even when V8 saves the receiver
between the initial load and the undefined check:

```text
ACCU = arg0.a
r8 = arg0
if (!(ACCU !== undefined)) { ACCU = 1 }
r0 = ACCU
```

The recovery keeps the simple receiver save and emits:

```js
r8 = arg0
r0 = (arg0.a === undefined ? 1 : arg0.a)
```

With that addition, `13_destructuring_spread` is now at `ACCU=0, reg_refs=42`
in vanilla-v8asm mode. The same cleanup also fixed a more general simplifier
hazard: register constants must not be inlined into the left side of compound
assignments, otherwise `r9 += 1` can become `2 += 1`. The bytenode row improves
only partially in the current default run because the forced-incompatible
object printer cannot reliably print the `"values"` string; it shows up as an
`<undefined: segmentfault...>` placeholder, so the spread matcher deliberately
does not guess.

The next cleanup targeted default-parameter bytecode and property self-updates.
V8 emits default parameters as an `ACCU !== undefined` branch, and class method
updates may then save a receiver property before adding the defaulted value:

```text
ACCU = arg0
if (!(ACCU !== undefined)) { ACCU = 1 } else { ACCU = arg0 }
r0 = ACCU
ACCU = this.n
r2 = this.n
ACCU = r0
ACCU = (r2 + ACCU)
this.n = ACCU
```

Level 4 now recovers this as:

```js
r0 = (arg0 === undefined ? 1 : arg0)
this.n = (this.n + r0)
```

This moved `15_class_method` from five `ACCU` assignments to one in both
vanilla-v8asm and bytenode modes, while preserving the value flow through the
saved register.

The remaining class and closure setup residue was the adjacent context creation
shape:

```text
ACCU = create_function_context(ScopeInfo_FUNCTION_SCOPE, 1)
r0 = pushContext(ACCU)
```

and its class-scope variant using `create_block_context(...)`. Level 4 now
collapses only this specific `pushContext(ACCU)` consumer:

```js
r0 = pushContext(create_function_context(ScopeInfo_FUNCTION_SCOPE, 1))
```

The rule is intentionally not a general "inline ACCU into any call" pass. It is
limited to the context push setup because the call is adjacent and the saved
register receives the previous context, not the accumulator value. This brings
the closure and class setup cases (`06_closure`, `09_all_features`, and
`15_class_method`) to zero ACCU assignment lines in vanilla-v8asm mode.

```text
17_async_await.js     async function lowering, suspend/resume generator state
18_private_fields.js  private field brand/name setup and keyed own property define
19_generator_yield.js plain generator function, yield, next(), resume-mode switch
```

These cases exposed translator fallbacks for:


Those opcodes now have explicit translations. The async generator instructions
are still represented as low-level pseudo operations rather than recovered
`await` syntax, but they no longer leak raw bytecode comments into level-4
output. After the opcode pass, cases 16-18 also report `unknown = 0`.

The plain generator case added later proved why `SwitchOnSmiNoFeedback` and
`Throw` also need explicit translations. V8 lowers generator resume-mode checks
into:

```text
ACCU = _GeneratorGetResumeMode(r0)
SwitchOnSmiNoFeedback ...
Throw
```

The decompiler now keeps that as a low-level state-machine comment instead of a
raw bytecode fallback:

```js
// SwitchOnSmiNoFeedback _GeneratorGetResumeMode(r0) [3], [2], [0] { 0: @39, 1: @36 }
throw r1
```

It also recovers the caller-side result selection:

```js
return (r4 + (r3.done ? r3.value : 0))
```

With those changes, `19_generator_yield` reports `unknown = 0`, `raw_goto = 0`,
and zero ACCU assignment lines in both vanilla-v8asm and bytenode modes. The
generator body is still intentionally low-level; recovering actual `function*`
syntax is a separate structuring problem.

The async case also exposed a correctness bug in register aliasing. Level 3
could cache `r2 = r4`, then later accept `r4 = r1.value` while still treating
`r2` as an alias of `r4`. That produced the wrong expression:

```js
r5 = (r4 + r4.value)
```

The base simplifier now invalidates cached register aliases when their source
register is reassigned, and the level-4 single-use inliner refuses to move an
alias across a write to any register referenced by the alias expression. The
same async case now keeps the intended two awaited values:

```js
r5 = (r1.value + r2.value)
```

The async resume-mode check also needed a narrower ACCU-compare recovery. V8
emits the check as:

```text
ACCU = 0
ACCU = (r5 === ACCU)
if (!(truthy(ACCU))) { ... }
```

Dropping the immediate `ACCU = 0` before rewriting the compare leaves the
misleading expression `(r5 === ACCU)`. Level 4 now rewrites that shape after
switch recovery, so the async output keeps the concrete resume mode:

```js
if (!(r5 === 0)) {
  throw r4
}
```

Ordering matters here. Running this compare compaction before the switch pass
breaks the two-case switch recognizer and reintroduces raw `goto` comments, so
the pipeline applies switch recovery first and only then compacts immediate
ACCU comparisons. A follow-up cleanup also collapses `ACCU = value; throw ACCU`
into `throw value`, which is common in async reject-mode branches.

The last async residue was not a normal expression at all. V8 places the async
reject handler after the resolve return, and the handler receives its thrown
value in the accumulator:

```text
return _AsyncFunctionResolve(r0, r5)
r5 = ACCU
SetPendingMessage
return _AsyncFunctionReject(r0, r5)
```

Level 4 now names that handler-entry value explicitly instead of pretending it
is an ordinary linear accumulator dependency:

```js
return _AsyncFunctionResolve(r0, r5)
r5 = async_reject_exception
// SetPendingMessage
return _AsyncFunctionReject(r0, r5)
```

After naming the handler exception, the earlier producer in the non-exception
path can be removed normally. With those rules, `17_async_await` is at zero
ACCU assignment lines in both vanilla-v8asm and bytenode modes, while
`08_switch` stays at `raw_goto=0`.

Optional chaining had a separate correctness trap. A keyed optional access such
as `tags?.[0]` was compacted from:

```text
ACCU = 0
ACCU = r2[ACCU]
r2 = ACCU
if (!(isNullish(ACCU))) { ... }
```

to only `r2 = r2[0]`, accidentally discarding the accumulator value needed by
the following nullish guard. The keyed-property compactor now preserves `ACCU`
when the next statement still reads it, and a new optional-chain recovery pass
collapses the nested nullish guards:

```js
r0 = (isNullish(arg0?.profile?.address?.city) ? "unknown" : arg0?.profile?.address?.city)
r1 = (arg0?.tags?.[0]?.length || 0)
```

This reduced `14_optional_chaining` from fifteen `ACCU` assignment lines to zero
in both vanilla-v8asm and bytenode modes.

Private fields exposed another translator-level correctness bug. V8 documents
`SetKeyedProperty` as:

```text
SetKeyedProperty <object> <key> <slot>
```

with the value in the accumulator. The earlier translator incorrectly used
`ACCU` for both key and value, producing output such as:

```js
this[ACCU] = arg0
```

The opcode now uses the key operand register, matching `DefineKeyedOwnProperty`.
Level 4 also handles the saved-key keyed-read shape:

```text
ACCU = context_slot[2]
r2 = context_slot[2]
ACCU = this[ACCU]
```

and recovers the private-field increment as:

```js
this[context_slot[2]] = (this[context_slot[2]] + r0)
```

This removes the misleading `this[ACCU]` output from `18_private_fields` and
reduces its residual `ACCU` assignment count. A later pass also handles the
private brand initializer guard:

```text
ACCU = closure[symbol]
if (!(ACCU === undefined)) {
  r1 = ACCU
  r1.call(this)
}
```

The accumulator producer is now inlined into both the equality guard and the
body reads, after unused `ACCU = call(...)` statements have been converted to
expression statements:

```js
if (!(closure[symbol] === undefined)) {
  r1 = closure[symbol]
  r1.call(this)
}
```

This brings `18_private_fields` to zero ACCU assignment lines in both
vanilla-v8asm and bytenode modes, while keeping the brand check explicit.

The regex/template case then improved the bound-call and string-concat passes.
Method calls wrapped by `String(...)` are now rewritten, so:

```js
r1 = String(r2.call(r3))
```

becomes:

```js
r1 = String(r3.toUpperCase())
```

The string concat cleanup also substitutes accumulator binary expressions inside
stringification:

```js
return (r1 + String((Number(r3) + 1)))
```

instead of keeping a separate `ACCU = Number(r3)` line. A narrow backward
liveness cleanup removes unused builtin aliases such as `r2 = Number` without
dropping effectful calls or local literal stores.

The same case also exposed a duplicated call/guard shape from `Call` followed
by `Star`:

```text
ACCU = r2.exec(arg0)
r0 = r2.exec(arg0)
if (!(truthy(ACCU))) { ... }
```

This is one runtime call in the bytecode, but the simplified pseudo-code had
both the accumulator producer and the saved register expression. Level 4 now
rewrites only the adjacent duplicate-store guard:

```js
r0 = r2.exec(arg0)
if (!truthy(r0)) { ... }
```

The rule intentionally accepts the wrapped-negation form only for this
duplicate-store pattern. Letting the more general `r0 = value; if
(!(truthy(ACCU)))` rewrite see that form breaks the two-case switch recognizer.
With the narrower split, `16_regex_template` is at zero ACCU assignment lines in
vanilla-v8asm mode, and `08_switch` remains at `raw_goto=0`.

The latest round case targets rest parameters plus spread calls and constructors:

```text
20_rest_spread_calls.js  CreateRestParameter, CallWithSpread, ConstructWithSpread
```

This first exposed a simple translator coverage gap. Without explicit handlers,
the three spread opcodes were left as raw bytecode comments:

```text
CreateRestParameter
CallWithSpread
ConstructWithSpread
```

`CreateRestParameter` is now emitted from the bytecode parameter count:

```js
r2 = Array.prototype.slice.call(arguments, 1)
```

and spread calls/constructors preserve the final spread register:

```js
r1 = new context_slot[3](...r0)
r3 = collect("x", ...r0)
```

The more important correctness bug was in the base simplifier. It cached
`r3 = []` as if the register stayed equal to the empty literal after later
stores:

```text
r3 = []
r3[0] = arg0
r3[1] = ACCU
r0 = r3
```

That allowed later spread sites to degrade to `...[]`, which is a real loss of
arguments rather than just noisy output. Property writes on register receivers
now invalidate cached aliases for the mutated receiver, and level 4 has a fixed
array-builder pass for the common `CreateArrayLiteral` plus `StaInArrayLiteral`
sequence. The recovered output keeps the actual values:

```js
r0 = [arg0, (arg0 + 1)]
r1 = new context_slot[3](...r0)
r3 = collect("x", ...r0)
r3 += Math.max(...r0)
```

Bound-call rewriting also needed a second pass after binary/string compaction,
because spread calls can appear inside compound assignments such as
`r3 += r4.call(Math, ...r0)`. That now recovers `Math.max(...r0)` when the
saved callee register is still known.

The current 20-case round has `ACCU = 0`, `unknown = 0`, and `raw_goto = 0` for
every vanilla-v8asm row. The same ACCU cleanup now also reaches zero in the
forced-incompatible bytenode rows by converting unused best-effort placeholder
property reads from `ACCU = obj["<undefined...>"]` to expression statements and
by inlining adjacent simple accumulator literals. This keeps the recovered read
visible without pretending the missing property name was decoded correctly.
Remaining `<undefined: segmentfault...>` placeholders in bytenode rows are
compatibility fallout, not proof that older 11.x/12.x bytenode variants were
tested.

# 0x5 Multi-Version Rule

For common V8 targets, build and identify `v8asm` by the full compatibility tuple, not just by version:

```text
V8 version string
code-cache magic
Version::Hash()
FlagList::Hash()
read-only snapshot checksum
pointer compression / sandbox build args
embedder suffix, if available
```

Node, Electron, and vanilla V8 may require separate builds even when their major/minor/build/patch values look close or identical.

The practical workflow is:

1. Use `checkversion` to identify the numeric V8 version hash.
2. Compare header compatibility against the current binary.
3. If `magic`, `flags_hash`, or `read_only_snapshot_checksum` differs, do not call it a supported match.
4. Build a matching target from the correct V8 branch/embedder configuration.
5. Use `--force-incompatible` only as a research fallback, never as normal output.

The round test runner now records this explicitly. `summary.md` starts with the
exact `v8asm` binary, `v8asm version`, build args, Node version, Node V8 version,
and bytenode version used for that run. It also writes per-case
`*.checkversion.txt` logs. If the Node V8 version does not match the `v8asm`
version, bytenode rows are forced-incompatible coverage only; they are not
evidence that a V8 major-version adaptation has a corresponding bytenode build.

There is now also a lightweight matrix runner:

```bash
./tests/decomp_rounds/run_version_matrix.sh
```

It uses one round case by default and writes:

```text
tests/decomp_rounds/version_matrix/summary.md
```

The matrix separates two questions that were easy to conflate:

1. Can each available `v8asm` binary compile, strictly disassemble, and
   decompile cache that it generated itself?
2. Does a bytenode cache generated by a local nvm Node version actually match a
   given `v8asm` binary closely enough to justify best-effort inspection?

The current local evidence is:

```text
v8asm self cache:
  repo-v8asm 13.6.233.10: ok
  out/v8asm.11.9.x64.release: 11.9.172, ok
  out/v8asm.12.4.x64.release: 12.4.254.21, ok
  out/v8asm.12.4.node22.x64.release: 12.4.254.21, ok
  out/v8asm.12.9.x64.release: 12.9.99, ok
  out/v8asm.13.4.x64.release: 13.4.114.21, rebuilt and usable for Atom with the matching Electron context snapshot

nvm nodes:
  Node v22.17.0: V8 12.4.254.21-node.26
  Node v24.7.0: V8 13.6.233.10-node.26
```

That means Node 22/bytenode is a 12.4 target, not a 12.9 target. It also means
the V8 build args matter, not only the version string. Node 22 reports
`process.config.variables.v8_enable_pointer_compression = 0`. A vanilla 12.4
`v8asm` built with `v8_enable_pointer_compression=true` has the same numeric V8
version, but forcing the bytenode cache into it crashes in
`CodeSerializer::Deserialize -> ObjectDeserializer -> ReadOnlySpace::AllocateRaw`.
This happens before bytecode object traversal or printing, so adding more print
guards cannot fix it.

The matrix now records both `numeric_match` and `pointer_match`. It tries
`--force-incompatible` by default only when the numeric V8 version matches and
the pointer compression layout is not known to differ. Numeric mismatches can
still be probed with `VERSION_MATRIX_FORCE_MISMATCH=1`, but signal-style exits
such as `fail:139` are hard failures, not warnings. With the Node-22-aligned
12.4 build:

```text
Node v22.17.0 / V8 12.4.254.21-node.26
  out/v8asm.12.4.x64.release:
    numeric_match=yes, pointer_match=no, force skipped
  out/v8asm.12.4.node22.x64.release:
    numeric_match=yes, pointer_match=yes, force_disasm=ok, decompile_force=ok
```

This is the current answer to the bytenode-version question: bytenode must be
checked against both the matching V8 major/minor revision and the compatible
embedder build layout.

The matrix is now a stricter CI-style gate instead of a passive report. In
addition to exit status and crash-signature checks, every successful level-4
output records:

```text
quality = raw:<n> unknown:<n> undef:<n>
```

The default thresholds require `raw=0` and `unknown=0`, so a structurer
regression or missing opcode translation fails immediately. `undef` remains a
reported count by default because it is often evidence for the Node/Electron
snapshot mismatch being investigated; set
`VERSION_MATRIX_MAX_UNDEFINED_FALLBACKS` when a focused run should fail on it.

For Node 24/bytenode versus repo `v8asm` 13.6, the numeric V8 version matches,
but strict deserialization still refuses the file:

```text
magic: 0xc0de0688 (expected 0xc0de0689) mismatch
flags_hash: 0x1c7c619b (expected 0xdc93751f) mismatch
read_only_snapshot_checksum: 0xd31c4342 (expected 0x436e38a3) mismatch
```

So even this row remains forced-incompatible research coverage, not a native
embedder match.

When switching V8 versions, follow the official V8 checkout flow. Do not just
move a worktree or swap source directories. Checkout the target revision in the
`fetch v8` checkout, then synchronize dependencies:

```bash
git checkout 13.4.114.21
gclient sync --with_branch_heads --with_tags
gn gen out/v8asm.13.4.x64.release --args='is_debug=false v8_enable_object_print=true v8_enable_disassembler=true v8_enable_pointer_compression=true'
autoninja -C out/v8asm.13.4.x64.release v8asm
```

Build output directories should be versioned and kept:

```text
out/node.x64.release
out/v8asm.11.9.x64.release
out/v8asm.12.9.x64.release
out/v8asm.13.4.x64.release
out/v8asm.13.6.x64.release
```

Do not run clean builds, remove old `out/*` directories, or use
`gclient sync -D` unless there is a concrete dependency conflict. V8 builds are
large, and preserving the GN/Ninja output directories is the practical way to
reuse prior compilation work when switching between nearby versions.


The 11.9 port needed real source adaptation, not only context refresh:

```text

ObjectBoilerplateDescription::capacity() -> length()
```

the same sandbox helper used by newer branches, so the 11.9 patch uses a
conservative read-only check before printing trusted objects. Both `v8asm` and
the patched d8 extension code were compiled in
`out/v8asm.11.9.x64.release`.
