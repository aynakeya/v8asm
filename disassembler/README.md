# Standalone V8 bytecode disassembler

The Python module accepts a complete V8 `ScriptCompiler::CachedData` buffer:

```bash
python3 -m disassembler input.jsc > /tmp/input.disasm.txt
```

Use `--version` when a custom runtime has an unknown version hash but its V8
source layout matches a checked-in profile. Use the startup snapshot whose
read-only checksum matches the cache:

```bash
python3 -m disassembler input.jsc \
  --version 13.4.114.21 \
  --snapshot-blob v8_context_snapshot.bin \
  > /tmp/input.disasm.txt
```

## Capturing the buffer accepted by Electron

An application can decrypt, decompress, unwrap, or rewrite a `.jsc` file before
passing it to V8. In that case, parsing the file from disk cannot recover the
runtime transformation. Load `capture_cached_data.cjs` before the application
loader to capture the `cachedData` value at the `vm.Script` boundary.

Linux or macOS:

```bash
V8_CACHE_DUMP_DIR=/tmp/v8-cache-dump \
NODE_OPTIONS="--require=/absolute/path/to/v8asm/disassembler/capture_cached_data.cjs" \
/path/to/electron-app
```

Windows PowerShell, using paths without spaces:

```powershell
$env:V8_CACHE_DUMP_DIR = "C:\tmp\v8-cache-dump"
$env:NODE_OPTIONS = "--require=C:\src\v8asm\disassembler\capture_cached_data.cjs"
& "C:\path\to\application.exe"
```

Each Electron process writes to its own PID directory:

```text
v8-cache-dump/
  12345/
    0000.input.jsc
    0000.normalized.jsc
    0000.payload.bin
    0000.json
```

- `*.input.jsc` is the exact buffer supplied to `vm.Script` after the
  application's JavaScript-side transformations.
- `*.normalized.jsc` is regenerated by the accepting Electron runtime through
  `script.createCachedData()`.
- `*.payload.bin` is the serializer payload extracted from the normalized cache
  without its V8 cached-data header.
- `*.json` records `process.versions.v8`, the source filename, sizes, and
  whether V8 rejected the cached data.

For example:

```json
{
  "pid": 12345,
  "sequence": 0,
  "v8_version": "13.6.233.10-node.26",
  "filename": "capture-test.js",
  "input_size": 544,
  "cached_data_rejected": false,
  "normalized_size": 536,
  "payload_offset": 32,
  "payload_size": 504
}
```

Payload extraction accepts a header layout only when its encoded payload length
exactly equals the bytes remaining after that header. If the layout is not
unique, `*.payload.bin` is omitted and both payload fields remain `null`; the
hook does not guess an offset.

Try the captured input directly first:

```bash
python3 -m disassembler 12345/0000.input.jsc \
  --version 13.2.152.41 \
  --snapshot-blob v8_context_snapshot.bin \
  > /tmp/main.disasm.txt
```

The filename and extension have no special meaning: if `*.input.jsc` contains a
standard V8 cached-data header, the normal disassembler path already supports
it. If its header remains private to a modified Electron build, use the
normalized output:

```bash
python3 -m disassembler 12345/0000.normalized.jsc \
  --version 13.2.152.41 \
  --snapshot-blob v8_context_snapshot.bin \
  > /tmp/main.disasm.txt
```

The matching snapshot is selected by read-only checksum, not filename. A cache
created in an Electron context commonly matches `v8_context_snapshot.bin`,
while a standalone V8 build commonly matches its own `snapshot_blob.bin`.

The hook is diagnostic and deliberately narrow. It observes `vm.Script`; an
application using a native addon or another V8 compilation API needs a capture
at that API boundary. Electron builds that disable the Node options environment
variable cannot preload this file through `NODE_OPTIONS`; instrument the
application's `.jsc` loader immediately around `new vm.Script(...)` instead.

### When `NODE_OPTIONS` is disabled

Some packaged Electron applications disable the
`EnableNodeOptionsEnvironmentVariable` fuse. The application may reject
`--require`, ignore `NODE_OPTIONS`, or start without producing a dump. Do not
add the capture script to the `.jsc` file itself; it must run before the loader
constructs `vm.Script`.

First try pausing the Electron main process before application code:

```powershell
$env:V8_CACHE_DUMP_DIR = "C:\tmp\v8-cache-dump"
& "C:\path\to\application.exe" --inspect-brk=9229
```

Open `chrome://inspect`, configure `localhost:9229`, attach to the paused main
process, and evaluate this in the debugger console:

```js
require(String.raw`C:\src\v8asm\disassembler\capture_cached_data.cjs`)
```

Resume execution only after the `require` call succeeds. This method does not
modify the packaged application. It is unavailable if the application also
disables the Node inspector or consumes the relevant cache before the main
process pause.

If the inspector is unavailable, instrument the JavaScript loader directly.
Work on a copy of the application and locate its archive, normally
`resources/app.asar`:

```powershell
Copy-Item "C:\path\to\resources\app.asar" "C:\tmp\app.asar"
npx @electron/asar extract "C:\tmp\app.asar" "C:\tmp\app-unpacked"
rg -n "cachedData|new vm\.Script|Module\._extensions.*jsc" "C:\tmp\app-unpacked"
```

In the matching loader, insert the absolute-path `require` before it imports or
copies `vm.Script`:

```js
require(String.raw`C:\src\v8asm\disassembler\capture_cached_data.cjs`);
const vm = require("node:vm");
```

If this is your application, run the extracted application in development or
rebuild it from the modified source. If ASAR integrity is not enabled, a test
copy can be repacked and installed into a copied application directory:

```powershell
npx @electron/asar pack "C:\tmp\app-unpacked" "C:\tmp\app.asar"
Copy-Item "C:\path\to\application-directory" "C:\tmp\application-test" -Recurse
Copy-Item "C:\tmp\app.asar" `
  "C:\tmp\application-test\resources\app.asar" -Force
```

Do not replace the original archive without keeping a backup. Applications
using embedded ASAR integrity or signature validation will reject a repacked
archive; in that case, add the one-line `require` in the application's source
before packaging, use the inspector method, or instrument the matching
Electron/V8 native cache-consumption path. Repacking cannot bypass an integrity
policy.

Launch the instrumented copy with only the output directory configured:

```powershell
Remove-Item Env:NODE_OPTIONS -ErrorAction SilentlyContinue
$env:V8_CACHE_DUMP_DIR = "C:\tmp\v8-cache-dump"
& "C:\tmp\application-test\application.exe"
Get-ChildItem "C:\tmp\v8-cache-dump" -Recurse
```

The capture succeeded when a PID directory contains `*.json` and
`*.input.jsc`. A false `cached_data_rejected` value means that V8 accepted the
captured input; `*.normalized.jsc` and `*.payload.bin` are then produced as
well.

## Parsing a raw serializer payload

The captured payload has no cached-data header:

```bash
python3 -m disassembler 12345/0000.payload.bin \
  --version 13.2.152.41 \
  --payload-offset 0 \
  > /tmp/main.disasm.txt
```

The same mode accepts output from a matching native Electron/V8 build that
dumps `SerializedCodeData::Payload()` directly:

```bash
python3 -m disassembler captured.bin \
  --version 13.2.152.41 \
  --payload-offset 0 \
  > /tmp/main.disasm.txt
```

The same option can address a payload embedded at a known offset in a larger
container:

```bash
python3 -m disassembler wrapped.bin \
  --version 13.2.152.41 \
  --payload-offset 0x20 \
  > /tmp/main.disasm.txt
```

This mode does not scan for a plausible stream. The offset must come from the
accepting runtime or the wrapper implementation. Because the raw payload omits
the cache header, the disassembler cannot validate its version hash, flags
hash, magic, or read-only snapshot checksum. `--version` is therefore required;
when supplying a snapshot, the caller must ensure it is the exact runtime
snapshot. Use `--runtime-variant` as well if the flags-specific runtime-ID table
is known.

If both captured files fail while the application accepts the cache, the
runtime likely contains a private V8 format or object-layout patch. In that
case, generate a profile from that exact source revision or dump the raw
serializer payload inside the accepting V8 build.
