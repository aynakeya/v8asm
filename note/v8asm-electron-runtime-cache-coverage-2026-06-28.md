# v8asm Electron runtime cache coverage

Date: 2026-06-28

## Why this note exists

The source build matrix proves that each patch can compile in Node-style and
Electron-style modes, but Electron-generated bytenode `.jsc` validation needs
real Electron release binaries. The Electron matrix must compile the `.jsc`
with the same Electron runtime whose `process.versions.v8` exactly matches the
selected Electron-flavored `v8asm`.

## Cache helper

Added:

```bash
python3 tests/decomp_rounds/fetch_electron_releases.py <version> [...]
```

The helper downloads official GitHub release zips into:

```text
/home/aynakeya/workspace/tmp/v8test/electron-cache
```

It preserves existing cache state:

- if an unpacked release already has `electron`, `snapshot_blob.bin`, and
  `v8_context_snapshot.bin`, it is reused;
- if the zip already exists, it is reused;
- existing unpacked directories are not deleted or overwritten.

The matrix still owns correctness: it reads each release's
`process.versions.v8` and selects only exact matching Electron-flavored cached
`v8asm` binaries.

## False-positive fix

The new Electron 35.7.5 runtime initially exposed a stale-cache issue:

```text
v8asm.13.4.electron.x64.release
```

had the exact V8 version string but did not expose
`v8_enable_static_roots` in `build-args`. The newer validated cache is:

```text
v8asm.13.4.114.21.electron.staticroots.x64.release
```

`check_electron_version_matrix.py` now ignores stale exact-match Electron
caches that lack explicit `v8_enable_static_roots=` metadata when a newer exact
match with that metadata exists. It still validates the selected exact match and
does not substitute nearby versions.
